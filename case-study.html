1. Introduction

Bubble is an open-source solution that automates the deployment of preview apps for dynamic frontend applications built with the Next.js framework.

Whenever a developer makes a pull request for a new UI/UX feature they are working on, Bubble will automatically build and deploy a fully-functioning draft of their web application—called a preview app—and provide a URL which can be shared with colleagues. This makes it easy for all relevant stakeholders to immediately review the changes and give feedback.

Bubble integrates with an application’s GitHub repository and automates the deployment and teardown of preview apps through a developer’s own AWS account, allowing developers to maintain full control over their source code and self-hosted cloud-based infrastructure. All preview apps can be easily managed through Bubble’s CLI tool and its user-friendly dashboard interface.

2. Brief History of Web Development

To better understand Bubble’s use case, we need to understand the web development pattern a typical Bubble user’s application would be built with. In order to do so, let’s first walk through a brief history of how web development approaches have evolved over time.

2.1 Server-Side Rendering

Traditionally, the most common strategy for building web pages and applications was based on client-server communication whereby a client, for example an end user’s browser, would request a page from a website or application, and the server would be responsible for responding with a fully-generated HTML page. With this approach, the server does most of the heavy lifting at request time - upon receiving the request from the client, the server would fetch any data from the database and other resources (i.e. CSS, images, etc.) needed in regards to the specific request, generate the full HTML components with the retrieved data and assets, and send this back to the client to display to the end user. This process, appropriately named server-side-rendering (SSR), would be repeated with every incoming request from the client.

2.2 Static Site Generation

As websites grew to serve more and more content, some of this content didn’t necessarily need to be completely “dynamic”, meaning there would be little to no change required for this content on every request. For example, consider a company landing page or a blog post, where the content of these pages can always stay the same regardless of who’s requesting it - it would be wasteful and inefficient to have the server dynamically generate these kinds of pages each time a client makes a request; rather, it’d be more ideal if an application could serve up the same, already generated HTML for every request. Web developers thus looked for a better, more lightweight approach to serve up these types of static pages, leading to the rise in popularity of static site generation (SSG).

With statically generated sites, complete HTML pages ready for serving to users are “pre-built” before the user ever requests them: at build time, a server is responsible for combining content and templates into ready-to-go HTML. This process of combining different files into HTML is very similar to the SSR process described above; the key difference lies in when the server carries out the combination: before the client ever makes a request.

The pre-built HTML, along with other assets needed to correctly display the page such as images and videos, is then typically cached in a content delivery network (CDN). A CDN is a geographically distributed group of servers that can quickly deliver content from a location closest to the client at request time.
